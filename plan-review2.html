🔴🟢🔵 本輪 TDD 開發流程

1

Red（紅燈）：先寫失敗案例

在 js/tests.js 對可自動化的項目（如對比度計算、CSP meta 是否存在、integrity 屬性是否存在）新增測試斷言，確認目前為紅燈；無法自動化的項目（鍵盤操作、CSP 實際攔截行為）改用情境式人工驗收條件。

2

Green（綠燈）：逐條最小修正

依 tasks.md 任務順序，一次只改一項（例如先修鍵盤事件，測過再修對比度色值），避免一次改動多檔難以定位問題來源。

3

Refactor（重構）：統一設計 token

待個別修正通過後，統一三頁 CSS 變數命名並收斂重複的色彩定義，確保後續調色只需改一處。

📋 任務清單（對應 sdd/ui-ux-security-review/tasks.md）

# 嚴重度 問題 修正動作
1 高 CSP worker-src blob: 可能未涵蓋 PDF.js worker 的 CDN 載入來源 瀏覽器 DevTools 實測是否被攔截，視結果調整 CSP 或 workerSrc 載入策略
2 高 拖曳上傳區 role="button" 缺少 keydown 處理 js/app.js 新增 Enter/Space 觸發選檔的 keydown 監聽
3 中 --text-muted (#6e7681) 對比度僅 4.125:1，未達 WCAG AA 三份樣式的 muted 文字色改為對比度 ≥4.5:1 的色值
4 中 plan.html／test.html 完全沒有 CSP 保護 依各頁實際載入資源補上對應 CSP meta 標籤
5 中 CDN 腳本只有 crossorigin，無 integrity SRI hash 四支 CDN 腳本加上正確的 integrity="sha384-..."
6 低 三頁各用一組不同名稱的 CSS 色彩變數 統一為 css/style.css 現有命名（--primary-color 等）
7 低 三頁皆無 favicon／theme-color 補上 inline SVG favicon 與 theme-color meta（本頁已示範做法）

✅ 驗收條件與情境

  • 鍵盤可用性：純鍵盤（Tab + Enter/Space）能完成「聚焦上傳區 → 觸發選檔對話框」。
  • CSP 不阻擋功能：三頁載入時 Console 無 CSP violation，且 PDF 上傳、預覽、ZIP 下載仍正常運作。
  • 對比度達標：調整後的 muted 文字色對背景對比度 ≥4.5:1（附計算結果）。
  • CSP 一致性：三頁 <head> 皆有對應 CSP meta 標籤。
  • SRI 落實：三頁 CDN <script> 皆有格式正確的 integrity 屬性。
  • 設計 token 統一：搜尋三份樣式，同語意色彩只用同一組變數名。
  • 行動裝置體驗：分頁圖示與網址列主題色正確顯示。
